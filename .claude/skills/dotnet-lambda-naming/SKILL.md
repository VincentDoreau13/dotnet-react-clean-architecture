# Lambda Naming Convention

Lambda parameters must be **self-documenting** everywhere. The reader must understand what the parameter represents without hovering over it in an IDE.

## Rule

**Never** use single-character (`c`, `e`, `x`, `p`, `t`, `i`, `o`, `r`, `s`, `n`) or generic opaque names (`item`, `obj`, `val`, `elem`).  
**Always** use the camelCase name that reflects what the parameter actually is.

## Examples by context

### EF Core configurations

```csharp
// BAD
builder.Property(c => c.Name).IsRequired().HasMaxLength(200);
builder.HasOne(c => c.Category).WithMany(cat => cat.Items);
builder.ToTable(t => t.HasCheckConstraint("CK_Price", "Price > 0"));

// GOOD
builder.Property(catalogItem => catalogItem.Name).IsRequired().HasMaxLength(200);
builder.HasOne(catalogItem => catalogItem.Category).WithMany(category => category.Items);
builder.ToTable(table => table.HasCheckConstraint("CK_Price", "Price > 0"));
```

### LINQ / collections

```csharp
// BAD
var names = items.Where(x => x.IsActive).Select(x => x.Name);
var total = orders.Sum(o => o.Amount);

// GOOD
var names = items.Where(catalogItem => catalogItem.IsActive).Select(catalogItem => catalogItem.Name);
var total = orders.Sum(order => order.Amount);
```

### FluentValidation

```csharp
// BAD
RuleFor(x => x.Name).NotEmpty();
RuleFor(x => x.Price).GreaterThan(0);

// GOOD
RuleFor(command => command.Name).NotEmpty();
RuleFor(command => command.Price).GreaterThan(0);
```

### Events, callbacks, delegates

```csharp
// BAD
button.Click += (s, e) => HandleClick();
services.AddSingleton(p => new MyService(p.GetRequiredService<IRepo>()));

// GOOD
button.Click += (sender, eventArgs) => HandleClick();
services.AddSingleton(provider => new MyService(provider.GetRequiredService<IRepo>()));
```

### MediatR / pipeline behaviors

```csharp
// BAD
await next();
_handlers.FirstOrDefault(h => h.CanHandle(request));

// GOOD — variable already named, lambda param reflects the type
_handlers.FirstOrDefault(handler => handler.CanHandle(request));
```

## Deriving the name

| What the parameter represents | Parameter name |
|---|---|
| `CatalogItem` entity | `catalogItem` |
| `Order` entity | `order` |
| `CreateCatalogItemCommand` | `command` |
| `GetCatalogItemQuery` | `query` |
| `IServiceProvider` | `provider` |
| `EntityTypeBuilder<T>` (table config) | `table` |
| Related navigation entity | use the related type name |

## Exceptions

The only acceptable short params are well-established **mathematical or algorithmic** conventions where the single letter *is* the name (e.g., `x`, `y` in geometric calculations, loop index `i` in a plain `for` loop). These do not apply to business/domain code.

## Exceptions

- **EF Core migrations** (`Migrations/` folder) — auto-generated by tooling, never edit manually and never flag.
- Mathematical/algorithmic conventions where the single letter *is* the name (`x`, `y` in geometry, loop index `i` in a plain `for` loop). These do not apply to business/domain code.

## Why

Clean Code principle — a name should reveal intent. Single-character lambda params force reliance on IDE tooling. Code is read far more often than it is written.
